Shared Cross-Client Rules
Authentication Strategy
All clients authenticate against Laravel API using Sanctum token-based authentication.
Backend returns a bearer token after successful login.
Frontend stores the token in a secure app session layer and never duplicates auth logic in every page.
Mobile stores the token in secure storage and injects it through a shared Dio interceptor.
Authorization Strategy
Authorization is enforced only by backend policies and scope filters.
Frontend and mobile may hide unavailable actions for UX, but backend remains the source of truth.
Permissions are returned in the auth profile payload and used only for presentation logic.
Time and Date Strategy
All timestamps are ISO 8601 in UTC.
Date-only values use YYYY-MM-DD .
Clients format dates locally for display only.Enum Strategy
All enum-like fields use snake_case strings.
Example: pending , approved , rejected , active , inactive .
Pagination Strategy
All list endpoints return the same pagination envelope.
Default per_page is 20 .
Maximum per_page is 100 .
Error Strategy
Validation errors always return HTTP 422 .
Forbidden actions always return HTTP 403 .
Missing resources always return HTTP 404 .
Unexpected failures always return HTTP 500 with no internal trace data exposed.
Backend Architecture Rules
Controller Rule
Controllers only map request to action and return API Resource responses.
Controllers must not contain business rules, direct query building, or status transition logic.
Validation Rule
Each write endpoint uses a dedicated Form Request.
Shared validators are allowed only for truly identical payloads.
Authorization Rule
Use Policies for entity-level authorization.
Use query scope filters for list visibility by branch and department.
Do not hard-code access rules in controllers.
Domain Rule
Business actions such as approve, reject, check-in, check-out, move stage, run payroll, and run report must be handled by dedicated action or service
classes.
Transaction Rule
Multi-table mutations must use database transactions.
Approval actions, balance adjustments, and payroll publication must always be transactional.
Response Rule
Always serialize responses through API Resources or DTO transformers.
Never return raw Eloquent models directly.
Audit Rule
Audit capture must occur in shared infrastructure or model observers where appropriate.
Controllers must not manually assemble audit payloads inline.

Core Backend ERD
Modeling Notes
Use Spatie tables for role and permission assignment.
Keep employee home assignment separate from branch and department access scope.
Prefer lean transactional tables with indexed foreign keys over duplicated summary tables.
Keep configuration-like rules in dedicated definition tables and only use JSON where flexibility is required.

Laravel Request and Resource Class Matrix
Naming Rule
Use Index<Domain>Request for filter-heavy list endpoints.
Use Store<Domain>Request and Update<Domain>Request for create and update endpoints.
Use <Action><Domain>Request for state transitions and one-off commands.
Use <Domain>Resource for single records and <Domain>Resource::collection(...) for paginated lists.
Use dedicated action classes for every approval, sync, publication, and workflow transition.

Seeder Environment Rule
Production seeders should include only baseline reference data, permissions, roles, and optionally one bootstrap super admin account.
Demo or QA data belongs in dedicated environment-gated seeders such as Database\\Seeders\\Demo\\... .
Payroll, attendance, leave requests, and other sensitive transactional data must not be seeded into production.
Permission Seed Matrix
Spatie Seeding Rule
Seed permissions first, then roles, then attach permissions to roles, then attach roles to employees.
Use a single guard name for API-authenticated users, for example sanctum or web , and keep it consistent across all seeded permissions and roles.
Treat permissions as the source of truth for backend authorization; roles are only bundles of permissions.
Prefer explicit permission strings in snake.case format exactly as documented by the API contract.
